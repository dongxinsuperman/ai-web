## 2. 数据库 DDL（PostgreSQL，库名 `aiweb`）

```sql
-- 批次
CREATE TABLE t_aiweb_submission (
    id            VARCHAR(32) PRIMARY KEY,          -- 短 uuid
    name          VARCHAR(255),
    callback_url  VARCHAR(1024),
    retry_max     INT NOT NULL DEFAULT 0,
    state         VARCHAR(16) NOT NULL DEFAULT 'accepted', -- accepted/done/cancelled/expired
    counts        JSONB DEFAULT '{}'::jsonb,         -- {success, failed, cancelled}
    summary_report_url VARCHAR(1024),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 执行单元（= 一条自然语言任务）
CREATE TABLE t_aiweb_item (
    id            VARCHAR(32) PRIMARY KEY,
    submission_id VARCHAR(32) NOT NULL REFERENCES t_aiweb_submission(id) ON DELETE CASCADE,
    case_id       VARCHAR(255),                      -- 调用方业务键（可作回放身份候选）
    case_name     VARCHAR(255),
    run_content   TEXT NOT NULL,                     -- 自然语言目标
    assets        JSONB DEFAULT '[]'::jsonb,         -- 本次引用的素材名列表
    state         VARCHAR(16) NOT NULL DEFAULT 'queued', -- queued/running/success/failed/cancelled
    status_reason VARCHAR(255),
    attempts      INT NOT NULL DEFAULT 0,
    retry_max     INT NOT NULL DEFAULT 0,
    report_url    VARCHAR(1024),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- 队列领取关键索引
CREATE INDEX idx_item_queue ON t_aiweb_item (state, created_at);

-- 单次执行记录
CREATE TABLE t_aiweb_run (
    id            VARCHAR(32) PRIMARY KEY,
    item_id       VARCHAR(32) NOT NULL REFERENCES t_aiweb_item(id) ON DELETE CASCADE,
    state         VARCHAR(16) NOT NULL DEFAULT 'running', -- running/success/failed
    steps         INT DEFAULT 0,
    token_usage   JSONB DEFAULT '{}'::jsonb,
    elapsed_ms    INT,
    fail_reason   TEXT,
    claimed_by    VARCHAR(64),                       -- 执行的 pod/worker 标识（多 Pod 预留）
    heartbeat_at  TIMESTAMPTZ,                       -- 心跳，用于超时回收
    started_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    finished_at   TIMESTAMPTZ
);
CREATE INDEX idx_run_item ON t_aiweb_run (item_id);
CREATE INDEX idx_run_heartbeat ON t_aiweb_run (state, heartbeat_at);

-- VLM 每步证据（也是将来 V2 回放的素材）
CREATE TABLE t_aiweb_run_step (
    id            BIGSERIAL PRIMARY KEY,
    run_id        VARCHAR(32) NOT NULL REFERENCES t_aiweb_run(id) ON DELETE CASCADE,
    step_no       INT NOT NULL,
    action        VARCHAR(32),                       -- click/type/scroll/.../finished/assert_fail
    thought       TEXT,
    action_detail JSONB,                             -- 解析后的统一动作对象（含坐标）
    screenshot_before VARCHAR(1024),
    screenshot_after  VARCHAR(1024),
    token_usage   JSONB,
    elapsed_ms    INT,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_step_run ON t_aiweb_run_step (run_id, step_no);

-- 素材库
CREATE TABLE t_aiweb_asset (
    id            VARCHAR(32) PRIMARY KEY,
    name          VARCHAR(255) NOT NULL,             -- 引用名（同名按最新覆盖或拒绝，见 §5.3）
    path          VARCHAR(1024) NOT NULL,            -- 存储后端中的路径/key
    size          BIGINT,
    mime          VARCHAR(128),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX uq_asset_name ON t_aiweb_asset (name);

-- 平台配置（热调）
CREATE TABLE t_aiweb_config (
    key           VARCHAR(64) PRIMARY KEY,           -- 仅 concurrency（缓存等为内核写死项，不入此表）
    value         VARCHAR(255) NOT NULL,
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 3. 状态机

```text
Submission:  accepted ──(全部 item 终态)──► done
                 │
                 ├──(整批取消)────────────► cancelled
                 └──(TTL 到期且仍有 queued)─► expired

Item:        queued ──(调度领取)──► running ──► success
                │                      └─────► failed ──(attempts<retry_max)──► queued(重试)
                └──(取消)──► cancelled

Run:         running ──► success / failed
```

终态判定（worker 写）：
- 内核 `finished` → run.success → item.success
- 内核 `assert_fail` 或抛异常 → run.failed → item.failed（若可重试则 item 回 `queued`，attempts+1）
- 内核 `call_user`（无人值守无法继续）→ run.failed，item.failed 且 `status_reason='needs_human'`（fail 类终态，单独标注，不自动重试）
- item 全部终态 → 汇总 submission.counts，submission.state=done，生成 `_summary.html`

## 5. 内核动作规范

### 5.1 统一动作对象

parser 把 VLM 文本输出解析成：

```python
{
  "action": "click",                 # 动作类型（click/left_double/right_single/drag/hotkey/type/scroll/wait/finished/call_user/hover/select_all_and_type/open_url/refresh/new_tab/switch_tab/close_tab/upload_file/assert_fail）
  "point": [500, 800],               # click/left_double/right_single/hover/scroll：0-1000 归一化坐标
  "start_point": [..], "end_point": [..],  # drag
  "direction": "down",               # scroll
  "content": "文本",                  # type / select_all_and_type / finished / assert_fail / call_user
  "key": "ctrl c",                   # hotkey：空格分隔小写、≤3 键
  "url": "https://...",              # open_url
  "index": 1,                        # switch_tab
  "name": "avatar.png"               # upload_file（引用素材）
}
```

### 5.2 动作集与 VLM 输出格式

**设计原则（重要）：承接，而非固定。**
我们**不规定**模型必须吐什么动作，只**承接**它吐出来的动作类型与格式——因为模型输出会随
豆包迭代 / 接入点 / 偶发别名而变化，写死白名单必然脆。因此：

1. **prompt 里给出推荐动作集**（下表，对齐豆包/UI-TARS 官方 `COMPUTER_USE_DOUBAO` 词表，
   豆包-seed-1.6 在该词表后训练过，用原生名 grounding 更准）——这是"引导"，不是"强约束"。
2. **parser 通用解析**：用统一正则抽 `fn_name(params)` + 参数（point/content/key/...），
   **不预设动作名**，任何函数式输出都能解析成统一动作对象（见 §5.1）。
3. **别名归一化层**：一张可扩展的 `ACTION_ALIASES` 表，把模型各种写法映射到 canonical 动作
   （如 `left_click`/`left_single`/`click`→`click`，`right_click`/`right_single`→`right_single`，
   `move`/`hover`→`hover`，`double_click`/`left_double`→`left_double`）。参考字节
   UI-TARS-desktop 的 `actionTypeMap` 做法。
4. **未知动作优雅兜底**：归一化后仍不认识的动作**不崩溃**——记录原始输出到 run_step、
   跳过该步并向模型注入"该动作不被支持，请换一种方式"的提示，让它自我纠正（必要时计入卡死）。
5. **靠测试反推真实格式**：用 §5.8 的动作探针测试集，拿**真实豆包输出**来校准别名表与解析，
   而不是凭空把格式固定死。

下表是"我们 canonical 承接 + Playwright 落地"的目标集；模型实际吐出的同义写法由别名层吸收。
坐标统一 `<point>x y</point>`（0-1000）。浏览器/工具类动作是我们内置的"选用型"动作（模型不对其做坐标 grounding）。

**A. 视觉键鼠动作（豆包原生词表，逐字对齐）**

| action | VLM 输出示例 | Playwright 落地 |
|---|---|---|
| `click` | `click(point='<point>500 800</point>')` | `mouse.click(x,y)`（含新标签页检测，自动 switch 到新 page） |
| `left_double` | `left_double(point='<point>x y</point>')` | `mouse.dblclick(x,y)` |
| `right_single` | `right_single(point='<point>x y</point>')` | `mouse.click(x,y, button='right')`（上下文菜单） |
| `drag` | `drag(start_point='<point>..</point>', end_point='<point>..</point>')` | `mouse.move→down→move→up` |
| `hotkey` | `hotkey(key='ctrl c')` | `keyboard.press('Control+C')`（空格分隔小写、≤3 键，映射到 Playwright 组合键） |
| `type` | `type(content='abc\\n')` | `keyboard.type('abc')`；content 以 `\n` 结尾则追加 `Enter` 提交（需先 click 聚焦） |
| `scroll` | `scroll(point='<point>x y</point>', direction='down')` | `mouse.wheel(0, dy)` / 容器 `evaluate scrollBy` 兜底 |
| `wait` | `wait()` | `sleep(5)` + 等页面稳定（豆包语义：等 5s 再截图） |
| `finished` | `finished(content='...')` | 终态：run 成功 |
| `call_user` | `call_user(content='需要人工：...')` | 无人值守：映射为 `needs_human` 终态（记 status_reason），等同 fail 但单独标注 |

**B. web 扩展视觉动作（非豆包原生，但实用，保留）**

| action | VLM 输出示例 | Playwright 落地 |
|---|---|---|
| `hover` | `hover(point='<point>x y</point>')` | `mouse.move(x,y)`（展开菜单/悬浮） |
| `select_all_and_type` | `select_all_and_type(content='abc')` | 便捷复合：全选+删除+输入（等价 `hotkey('ctrl a')`+`type`，保留以降模型负担） |

**C. 浏览器/工具内置动作（选用型，非 grounding；取代旧 `platform_tool` 反射）**

| action | VLM 输出示例 | Playwright 落地 |
|---|---|---|
| `open_url` | `open_url(url='https://...')` | `page.goto(url)` |
| `refresh` | `refresh()` | `page.reload()` |
| `new_tab` | `new_tab()` | `context.new_page()` |
| `switch_tab` | `switch_tab(index='1')` | 切到 `context.pages[index]` |
| `close_tab` | `close_tab()` | `page.close()` + 切回最后一个 |
| `upload_file` | `upload_file(name='avatar.png')` | 见 §5.3 |
| `assert_fail` | `assert_fail(content='...')` | 终态：run 失败（断言不通过） |

> 取消 `platform_tool` 反射；导航/刷新/标签/上传全部内置为 C 组原生动作。
> 终态共三种：`finished`（成功）/ `assert_fail`（失败）/ `call_user`→`needs_human`（需人工）。

### 5.4 坐标转换

Playwright **显式设定 viewport**（如 `1280×800`，可配），VLM 坐标统一换算：

```text
x = clamp(vlm_x / 1000 * viewport_w, 0, viewport_w-1)
y = clamp(vlm_y / 1000 * viewport_h, 0, viewport_h-1)
```
viewport 固定可控 → 不再需要标准版"注入锚点 div"那套 hack。注意 `device_scale_factor` 与截图尺寸保持一致。

### 5.5 决策循环（移植 ai_vlm_web_segment）

```text
init: previous_response_id=None, messages=[], segments=1, system_prompt=build(goal)
for step in 1..SAFETY_MAX_STEPS(100):
    # 分段重置：上一轮 prompt_tokens ≥ 阈值(默认30000) 且已有会话 → 断会话+注入续接提示
    if prev_prompt_tokens >= RESET_THRESHOLD and previous_response_id:
        previous_response_id=None; segments+=1; inject(resume_hint)
    wait_page_stable()
    shot = screenshot()
    decision = vlm_decide(shot)            # Responses API，见 §5.6
    thought, action_str = parse(decision)
    parsed = parse_action(action_str)
    record_step(...)                       # 落库 run_step + 截图（报告/回放证据）
    if action == assert_fail: run.failed; raise
    if action == call_user:   run.failed(status_reason='needs_human'); return
    if action == finished:    run.success; return
    execute(parsed)                        # Playwright
    screenshot_after()
    check_click_stuck(); check_scroll_stuck()
raise 安全上限
```

### 5.6 VLM 调用（主动式上下文缓存）

```python
POST https://ark.cn-beijing.volces.com/api/v3/responses
{
  "model": "<接入点>",           # 默认 doubao-seed-1-6-vision 接入点
  "temperature": 0,
  "input": [ {system}(仅首轮), {user: [hints..., image, "next step?"]} ],
  "caching": {"type": "enabled"},
  "store": true,
  "thinking": {"type": "disabled"},
  "previous_response_id": "<上一轮 id，非首轮>"
}
# 读取 usage.input_tokens / output_tokens / cached_tokens，累计并落库
```
缓存为**内核写死逻辑**（默认启用、分段阈值固定 30000），**不对外暴露、不可配置**；仅作为内核常量存在。

### 5.7 卡死检测（V1 朴素版）

- 连续 4 次点击近似同坐标（±30px）→ 注入提示，清空记录。
- 连续 3 次同方向滚动 → 注入提示。
- 阈值常量：`CLICK_STUCK_THRESHOLD=4`，`SCROLL_STUCK_THRESHOLD=3`。

### 5.8 动作探针测试集（用真实豆包输出校准，不预先固定格式）

目的：与其凭空规定豆包"应该"吐什么，不如**让豆包在受控场景下真吐一遍**，把原始 `Action`
字符串抓下来，反推出它当前真实的动作名与参数格式，再据此校准 `ACTION_ALIASES` 与 parser。

做法：

- **每类动作准备一个最小诱发场景**（一个目标页 + 一句 runContent），让模型必然产出该动作：
  - 单击 / 双击（双击打开 / 选中）/ **右键（呼出上下文菜单）** / 拖拽（滑块 / 排序）
  - **组合键**（全选 ctrl a、复制 ctrl c、粘贴 ctrl v）
  - 输入（含 `\n` 提交）/ 滚动（长列表）/ 悬停（展开菜单）
  - 导航 / 刷新 / 新开切换关闭标签 / **上传文件** / 等待
  - 终态：finished / assert_fail / call_user（给一个无法完成的目标）
- **采集**：内核已逐步落库 `t_aiweb_run_step.action_detail` 与原始 `Action` 文本；
  探针跑完后导出"原始输出样本集"。
- **校准**：对照样本里出现的实际动作名 / 参数键 / 坐标格式，补全别名表、修正正则；
  发现新动作类型 → 决定"映射到已有 canonical"或"新增一个 canonical + executor 落地"。
- **回归**：该测试集纳入 CI/手动回归；豆包接入点或模型升级后重跑，**第一时间发现格式漂移**。

> 原则复述：parser/别名层是"可演进的承接层"，探针测试集是"持续校准它的事实来源"。
> 任何"模型输出变了"的情况，应表现为"别名表加一条 / 正则宽一点"，而不是线上崩溃。

## 6. 调度与并发

### 6.1 原子领取（多 Pod 安全）

```sql
UPDATE t_aiweb_item
SET state='running', updated_at=now()
WHERE id = (
  SELECT id FROM t_aiweb_item
  WHERE state='queued'
  ORDER BY created_at
  FOR UPDATE SKIP LOCKED
  LIMIT 1
)
RETURNING *;
```

### 6.2 dispatcher 循环

```text
loop:
    N = config.concurrency           # 热读
    while running_count < N and 有空闲槽:
        item = claim_one()           # §6.1
        if not item: break
        run = create_run(item, claimed_by=POD_ID, heartbeat=now)
        spawn worker(item, run)      # asyncio task / 线程
    sleep(poll_interval)
```

### 6.3 worker

```text
context = browser.new_context(viewport=..., ...)
try:
    runner.run(item.run_content, item.assets, on_step=record_step, heartbeat=touch)
    mark success
except AssertFail/Exception:
    mark failed; 若 attempts<retry_max → item 回 queued
finally:
    build_report(); fire_webhook(item.terminal)
    context.close()
    if submission 全终态: summary + webhook(submission.terminal)
```

### 6.4 心跳与回收（reaper）

- worker 执行中周期 `UPDATE t_aiweb_run SET heartbeat_at=now()`.
- reaper 定时把 `state='running' 且 heartbeat_at < now()-TTL` 的 run 标 failed，对应 item 回 `queued`（防 Pod 崩溃后任务卡死）。

### 6.5 重启恢复（V1 简化）

启动时把本 `POD_ID` 名下 `running` 的 run 标 failed、item 回 queued（本 Pod 残留回收）；跨 Pod 由 reaper 兜底。

## 8. 自包含 HTML 报告结构

单 item 一个 `report.html`，离线可看：

- 头部：caseId / caseName / runContent / 终态 / 耗时 / 步数 / token 与缓存命中 / 估算成本。
- 步骤时间线：每步卡片 = 步号 + 动作 + Thought + 决策截图 + 执行后截图 + 耗时 + token。
- 终态块：finished/assert_fail 的 content；失败原因。
- 批次 `_summary.html`：item 列表 + 各自 reportUrl + 计数。
- 可选：附 Playwright Trace 文件下载链接（深度排障）。
