# 开源边界与安全

AI Web 开源的是浏览器自动化执行能力，不包含任何部署者的模型 Key、数据库、站点账号、Cookie、报告、截图或私有基础设施。

## 不会进入 Git 的本地数据

项目的 `.gitignore` 已排除：

- `backend/.env`：数据库地址、模型 Key、API Token 等真实配置。
- `backend/data/` 和 `data/`：报告、截图、素材等运行数据。
- `node_modules/`、`dist/`、虚拟环境和缓存。

公开仓库前仍应检查 Git 历史，而不是只检查当前目录。不要手工把 `.env`、数据库导出、报告压缩包或真实登录态添加到 Git。

## 部署要求

默认空的 `AIWEB_API_TOKEN` 只适合本机开发或受控内网。面向公网或不可信网络部署时，至少应做到：

1. 设置强随机的 `AIWEB_API_TOKEN`。
2. 使用 HTTPS/WSS 和受控反向代理。
3. 只允许受信的 Agent 连接 Server。
4. 将 PostgreSQL、报告目录和素材目录放在受控网络与持久化存储中。
5. 对上传文件、回调地址、站点登录接口配置建立部署侧访问控制。

AI Web 不是多租户公有 SaaS。不要把一个匿名 Server 直接暴露在互联网，再让任意人提交浏览器任务、上传素材或配置登录态。

## 登录态与站点配置

站点免登功能可以保存 Cookie、Playwright storageState 或登录接口配方。这些信息可能等同于账号访问凭据。

- 当前实现会将站点配置保存在数据库中，**不提供字段级加密**。
- 不要把生产 Cookie、账号密码或 token 写进任务文本、README、截图、日志或 Git。
- 只让受信的 Server、数据库管理员和 Agent 使用这些数据。
- 为不同环境使用独立测试账号和独立登录态；失效或泄露时立即在目标系统侧撤销。

## Function Map Context 与报告

Function Map Context 常包含测试账号、业务入口、验证码规则或内部术语。它会下发给执行 Agent，但默认不写入 Webhook；仍应按敏感配置处理。

报告和截图可能包含页面内容、用户资料或业务数据。生产环境需要设置保留周期、访问权限和清理策略，不能把报告目录直接公开为匿名文件盘。

## 发布前检查

- 确认 `git status --ignored` 中的 `.env`、数据目录没有被强制加入版本控制。
- 扫描当前文件和 Git 历史中的 Key、token、私有域名、数据库连接、账号和截图。
- 使用脱敏的 example.com、演示账号和示例 JSON。
- 确认 [MIT 许可证](../LICENSE)、[贡献指南](../CONTRIBUTING.md) 与 [安全策略](../SECURITY.md) 符合发布方式；公开前在 GitHub 启用 Private vulnerability reporting。
- 确认你拥有全部代码、图标、截图和演示素材的公开权利。
