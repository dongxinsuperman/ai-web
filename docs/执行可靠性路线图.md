# 执行可靠性路线图

AI Web 的目标不是只让模型“能点网页”，而是逐步形成和 AI Phone 相近的**可监督执行链路**：任务为什么成功、为什么失败、是否走偏、能否安全复跑，都应当有证据和明确边界。

但 AI Web 不会照搬 AI Phone 的手机设备、驱动、唤醒和真机管理能力。它会结合 Playwright、浏览器状态、截图和 Web 页面特征，做浏览器原生的实现。

## 当前已经具备

| 能力 | 当前状态 | 作用 |
|---|---|---|
| 页面稳定等待 | 已实现 | 等待页面加载和 DOM 短暂静默后再截图决策，减少模型看到过渡画面 |
| 结构化任务识别 | 已实现 | 识别含“操作步骤 / 预期结果”等标签的任务，决定是否启用最终二次断言 |
| 最终二次断言 | 已实现 | 结构化任务宣告完成后，用独立模型结合截图和步骤摘要复核结果 |
| 基础卡死提示 | 已实现 | 连续点击近似坐标或持续同向滚动时，向主模型注入换策略提示 |
| 步骤证据记录 | 已实现 | 保存截图、Thought、动作、耗时和 Token，用于报告与排障 |

这些能力说明 AI Web 已经不完全依赖模型自述；但它目前仍是精简版本。

## 计划逐步靠近的能力

### 1. 子步骤系统

计划让结构化任务在起跑时把“操作步骤”拆成有序子步骤，并让 Agent 在每轮判断：

- 当前子步骤是否已完成；
- 截图中有什么完成证据；
- 应继续当前步骤，还是合理跳过并进入下一步。

例如任务是“进入订单页、筛选已退款、打开第一条订单、确认退款状态”。当前模型可能看到显眼订单就提前打开；子步骤系统会要求它先确认“已退款筛选”是否完成，避免跳步或重复点击。

**状态：计划中，尚未实现。** 当前报告不会展示子步骤进度，也不会因为跳过子步骤而自动终止任务。

### 2. 审判系统

计划在出现异常信号或执行到一定步数时，调用独立的轻量模型审视最近动作与任务目标，输出“继续 / 修正 / 终止”的保守判断。

它主要解决两类问题：

- 模型不断操作但页面没有推进；
- 模型动作看似顺畅，却绕过了结构化任务中的关键步骤。

**状态：计划中，尚未实现。** 当前 AI Web 只有本地提示，不会在中途启动独立审判模型。最终二次断言也不能替代中途审判，因为它只在模型宣告完成后发生。

### 3. 增强卡死检测

计划从“连续相近点击 / 连续滚动”的简单提示，升级为综合判断：页面视觉状态是否反复出现、滚动是否无进展、动作是否来回震荡、未知动作是否持续累积。

**状态：计划中。** 当前基础检测只会提醒主模型换策略，不会直接判失败。

### 4. 安全回放与恢复

成功任务未来可以沉淀为可复用轨迹，但不会直接重放旧坐标。浏览器版会优先考虑：动作意图、当前页面状态、可访问性信息或定位线索，以及截图证据。

当页面状态不匹配时，应等待、局部修复或回退到 VLM 决策，而不是盲目继续点击。

**状态：计划中，尚未实现。** 当前 `cacheMode` 为兼容调用方而接受，但不会启用轨迹缓存。

## 实施原则

1. **先证据、后自动化。** 每个新增判断都要能在报告中解释触发原因和依据。
2. **默认保守。** 判断不充分时优先提示、记录或回退，不轻易把正常任务误杀。
3. **浏览器原生。** 优先利用 Playwright 和网页状态，不把真机坐标回放方案原样照搬。
4. **渐进启用。** 新能力先在结构化任务中启用，保持一句话自由任务的低门槛。
5. **明确降级。** 辅助模型、子步骤解析或缓存不可用时，报告和回调应能暴露“本次按降级模式执行”。

## 不在当前计划内

- 手机真机驱动、设备锁、应用安装、唤醒和镜像能力；这些属于 AI Phone。
- 将 AI Web 变成项目/用例/需求管理平台；这些属于上游工作台。
- 无限制的自动修复或静默重试；失败应保留证据并由调用方决定后续处理。
