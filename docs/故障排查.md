# 故障排查

## Server 正常启动，但任务一直 `queued`

最常见原因是没有可用 Agent。

检查：

1. Agent 进程是否仍在运行。
2. Agent 的 `--agent-id` 是否出现在 `AIWEB_BROWSER_SLOTS` 中。
3. 对应浏览器的容量是否大于 0。
4. 提交的 `platforms` 是否与可用容量匹配。

例如配置是 `{"mac-01":{"chrome":1}}`，但 Agent 使用 `--agent-id win-01`，Server 不会把任务派给它。

## Agent 连不上 Server

- 确认 `--server` 是 Agent 机器可访问的地址，而不是 Agent 自己的 `127.0.0.1`。
- 开启 `AIWEB_API_TOKEN` 后，Agent 也必须使用同一个 token。
- 经由反向代理时，确认 `/api/browser-agents/ws` 支持 WebSocket Upgrade，并且 HTTPS 对应 WSS。
- 检查 Server 的 `/health` 是否可从 Agent 机器访问。

## 报告 URL 是 127.0.0.1，其他人打不开

设置 Server 的 `AIWEB_PUBLIC_BASE_URL` 为调用方可访问的完整地址，例如 `https://aiweb.example.com`。报告和素材 URL 都由此值生成。

## Agent 报浏览器不存在

在 Agent 机器执行：

```bash
python -m playwright install chromium firefox webkit
```

只安装了 Chromium 时，只能提交 `chrome`；提交 Firefox 或 WebKit 会因环境缺失失败。

## 模型调用失败

检查 `.env` 中的：

- `AIWEB_VLM_PROVIDER`
- `AIWEB_VLM_BASE_URL`
- `AIWEB_VLM_API_KEY`
- `AIWEB_VLM_MODEL`

辅助模型字段留空时会按同一家模型配置回退；主模型本身缺少 Key、URL 或模型名时，任务无法开始决策。

## 有头模式无法运行

无显示器的 Linux Server 或容器应使用 `AIWEB_HEADLESS=true`。需要观察浏览器时，再把 Agent 放在有图形桌面的机器上并设置 `AIWEB_HEADLESS=false`。

## 登录态没有生效

- 确认站点关键字确实命中当前任务文本。
- 检查 Cookie 域名、storageState 的 origins 和登录接口配方。
- 使用独立测试账号；过期 Cookie 和 token 需要重新生成。
- 登录态页面验证需要至少一个在线 Agent。

## 任务完成却没有 Webhook

Webhook 是 best-effort：确认 `callbackUrl` 是 Server 可访问的 `http(s)` 地址，并检查调用方是否返回成功响应。报告和任务状态会先落库；Webhook 失败不会回滚执行结果。
