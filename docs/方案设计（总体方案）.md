## 1. 一句话定位

> **AI Web 是一个全新的、独立的小项目——"Web 版的、精简的 ai-phone"。**
> 用户提交一段自然语言目标，平台用 VLM 视觉大模型驱动浏览器把任务跑出来，
> 产出自包含报告并通过 Webhook 回调结果。

它**不是**测试平台，**没有**用例 / 项目 / 模块 / 步骤这些概念，只有：

- **队列**（提交进来的自然语言任务排队）
- **并发执行**（多个浏览器同时跑）
- **执行记录 + 报告**
- **素材文件管理**（执行中要上传的文件，统一增删改查）
- **对外接口 + Webhook 回调 + CLI**

UI 是一个**运维 / 维护台**，不是业务编排平台。

## 2. 与现有系统的关系

| 维度 | 现有 Web 自动化平台（标准模式） | AI Web |
|---|---|---|
| 定位 | 测试平台（项目/用例/步骤/测试集） | 自然语言执行中台（Web 版） |
| 浏览器 | 远程 Selenium Grid（4 个 standalone-chrome 容器，VNC 监控） | 本地内置浏览器（按需启动，无 Grid、无 VNC） |
| 执行 | `subprocess + pytest + allure` | 常驻调度器 + worker 池，直接驱动内核 |
| 框架 | Selenium | **Playwright** |
| 数据 | MySQL，`t_web_*` 系列表 | **PostgreSQL，独立库 + `t_aiweb_*` 系列表** |
| 代码 | 现有仓库 | **全新独立仓库 / 项目** |

**关键原则：抄逻辑，不抄框架。**

- 内核**决策 + 缓存逻辑**以标准项目最新内核 `webtest/keyswords/ai_vlm_web_segment.py` 为蓝本（见 §7）。
- 动作能力清单参考 `webtest/tool/webrun/extend/keywords.py`（点击/输入/滚动/拖拽/上传/导航/切关窗/按键/断言…）。
- 其余（Flask Controller/Model 风格、pytest+allure、Grid 远程、反射式 `platform_tool`）**全部不要**，技术栈在新项目里重新拍板。

## 3. 为什么砍掉 Grid，改成"单服务 + 本地浏览器"

现有约束：①没有多余资源再起独立 Grid 容器；②不能影响"很多人在用"的标准模式。

根因都来自"和标准模式共享同一批远程 Grid 浏览器"。已确认现有 Grid 形态（标准平台 `docker-compose.yml`）：

```yaml
image: selenium/standalone-chrome
environment:
  - SE_NODE_MAX_SESSIONS=1                  # 每个容器同一时刻只能 1 个 session
  - SE_ROUTER_NEW_SESSION_WAIT_TIMEOUT=3600 # 节点忙时新请求阻塞排队最多 1 小时
```

- 单个 Grid 服务并发上限 = 1；忙时新连接阻塞排队（"指定浏览器忙了会一直等"的根源）。
- 若 AI Web 共用这 4 台，会和标准模式互抢，踩中"不影响标准模式"的底线。

**结论（已确认）：AI Web 不走远程 Grid，把浏览器（本地 Chromium，Playwright 内置）直接打包进 AI Web 服务进程内。**

收益：

- 标准模式**一行代码不用改**，物理隔离、零影响。
- 不需要新增任何 Grid / VNC 基础设施。
- **"忙/闲调度 / 挑浏览器 / 排队"复杂度直接塌缩为一个并发信号量**：服务里开 N 个 worker，每个 worker 自己起一个本地浏览器跑任务，槽位占满就在队列里等。aiphone 那套设备锁/别名池在 Web 这边完全不需要。

代价（已接受）：放弃 VNC 看过程（只看报告）；并发上限由部署资源决定。

## 4. 总体架构

```text
                 ┌─────────────────────────────────────────────────────┐
 外部平台 / CI    │                  AI Web 后端服务（单进程/可多 Pod）       │
 / CLI / 人工 ──▶ │  FastAPI:  POST /api/submissions  GET 查询  取消  Webhook │
                 │      │                                                   │
                 │      ▼                                                   │
                 │   提交落库(PG: t_aiweb_*)  ─►  队列（DB 行锁领取）         │
                 │      │                                                   │
                 │      ▼  Scheduler（并发信号量 N，可热调）                  │
                 │   ┌─ worker_1 ─ 本地 Chromium(context) ─ VLM 内核 ─ 报告  │
                 │   ├─ worker_2 ─ 本地 Chromium(context) ─ VLM 内核 ─ 报告  │
                 │   ├─ ...                                                 │
                 │   └─ worker_N                                            │
                 │      │                                                   │
                 │      ▼  自包含 HTML 报告(共享存储) + 终态广播/回调          │
                 └─────────────────────────────────────────────────────┘
                          ▲
                          │ REST（前后端分离，独立部署）
                 ┌──────────────────────────────────────┐
                 │   AI Web 前端（维护台）                  │
                 │  队列总览 / 执行记录 / 报告 / 素材 / 配置   │
                 └──────────────────────────────────────┘
```

形态：**一个后端服务（API + 队列 + 调度 + 本地浏览器 + 内核 + 报告 + 素材）+ 一个前端维护台**，前后端分离、各自独立部署。

## 5. 技术栈选型

| 层 | 选型 | 说明 |
|---|---|---|
| 后端框架 | **FastAPI**（Python，async） | 与 ai-phone 一致；async 适配队列 / 调度 / Webhook |
| 浏览器执行 | **Playwright（Python）** | 见 §6，上传 / 并发 / 能力全面优于 Selenium |
| VLM 调用 | 火山方舟 **Responses API + 显式缓存**（主动式缓存） | 沿用标准项目最新内核机制，见 §7 |
| ORM | **SQLAlchemy 2.0** | 成熟，async 支持 |
| 数据库 | **PostgreSQL**（复用现有本地实例，新建独立库 `aiweb`） | 已确认；公司版后续同样新建库，不另搭 |
| 任务调度 | asyncio 事件循环 + worker 池 + **DB 行锁领取队列** | 主流、轻量、无外部中间件；天然多 Pod 安全（见 §8） |
| 文件 / 报告存储 | 共享存储（V1 本地卷；多 Pod 时换对象存储 / NFS） | 报告与素材需跨 Pod 可访问 |
| 报告 | **自绘自包含 HTML 为主 + 可选挂 Playwright Trace** | 见 §10 |
| 前端 | **Vue3 + Vite + Element Plus + Pinia** | 团队标准栈 |
| 打包 | Docker，基于 `mcr.microsoft.com/playwright/python` | 浏览器已内置，自包含 |

> V1 不引入 Redis / Kafka；队列与锁用 PG 实现。

## 6. 执行框架：为什么是 Playwright（而不是 Selenium）

核心诉求：能力尽量全、能传图 / 传文件 / 带工具、支持"配置文件→执行时自动抓取上传"、并发强。

| 诉求 | Playwright | Selenium |
|---|---|---|
| 上传文件 / 图片 | `set_input_files()` 直接喂隐藏 `<input type=file>`；`expect_file_chooser()` 接管弹窗式上传，两种场景全覆盖 | 仅能操作可见 input，弹窗上传困难 |
| 配置文件→自动抓取上传 | upload 动作从"素材库"取文件喂 `set_input_files`，天然契合 | 实现别扭 |
| 能力丰富 | 网络拦截 / 下载管理 / CDP / JS 注入 / 多标签 / iframe / shadow DOM 内置 | 多需自拼 |
| 并发 | **单进程内多个隔离 `browser_context` 并发**，轻量 | 每并发一套 chromedriver+Chrome 进程，重 |
| 报告 | 自带 **Trace Viewer**（每步截图+DOM快照+网络），可作深度排障件 | 需自绘 |
| 自包含部署 | `playwright install chromium firefox webkit` 一条命令装齐三引擎 | 驱动 / 版本要对齐 |
| 多浏览器 | **同一套内核驱动 chromium/firefox/webkit**，`platform` 切引擎零成本 | 各浏览器各配 driver，繁琐 |
| VLM 坐标动作 | `mouse.click(x,y)` / `mouse.wheel` / `keyboard.type` 与动作集一一对应，更简洁 | 靠 ActionChains + 注入锚点 div 的 hack |

**内核迁移成本可控**：与框架无关的部分（决策循环、缓存逻辑、解析、坐标换算、卡死检测、finished/assert_fail、system prompt）直接照搬；只需重写 `_execute_action` 那十几个动作（Playwright 版更短更稳）。

## 7. 执行内核设计（VLM Kernel）

**蓝本**：标准项目最新内核 `ai_vlm_web_segment.py`（Responses API + 显式缓存 + 分段）。
移植策略 = **决策/缓存逻辑照搬 + 执行层换 Playwright + 砍反射 + 内置动作**。

### 7.1 决策闭环

```text
[分段重置检查] → 页面稳定等待 → 截图 → 喂 VLM(Responses API) → 解析 Thought+Action
  → 坐标换算 → Playwright 执行 → 记录前/后截图 → 卡死检测 → 循环
（直到 finished / assert_fail / 安全上限 SAFETY_MAX_STEPS）
```

### 7.2 主动式缓存机制（保留并预留）

> 注意区分两种"缓存"，本项目说的是第 1 种：
> 1. **LLM 上下文缓存（主动式 / 显式缓存）**——本内核自带，省 token / 算力。**V1 保留启用。**
> 2. **执行回放缓存（成功 Run 回放）**——更大特性，**V1 不做，架构预留**。路线见 §7.4。

LLM 上下文缓存照搬自 `ai_vlm_web_segment.py`：

- 调用火山方舟 **Responses API**（`/api/v3/responses`），`caching:{type:enabled}` + `store:true` + `previous_response_id` 串会话；服务端维护全量历史，KV 命中跳过 prefill。
- 客户端每轮只发**增量**（注入提示 + 截图 + 默认问题），首轮才带 system。
- **分段重置**：上一轮 `prompt_tokens ≥ 阈值(默认30000)` 时，下一轮断开 `previous_response_id` 重起会话，并注入"会话续接"提示，把长任务切成多段，每段都留在最便宜计费档。
- 统计 token / 缓存命中率 / 估算成本，写入执行记录与报告。
- **可配置开关**：缓存阈值、是否启用，作为内核参数预留；将来要接轨迹回放，也在此层扩展。

### 7.3 动作集（全部内置为原生动作，砍掉反射式 platform_tool）

视觉类（0-1000 归一化坐标 → Playwright `mouse`/`keyboard`）：

- `click` / `double_click` / `hover`
- `type` / `select_all_and_type`
- `key_press`
- `scroll(direction)`
- `drag(start, end)`
- `wait`

浏览器 / 工具类（原来靠关键字反射，现在内置）：

- `open_url` / `goto`（导航）
- `refresh`
- `new_tab` / `switch_tab` / `close_tab`
- `upload_file`（核心，对接素材库，见 §11）
- 终态：`finished(content)` / `assert_fail(content)`

VLM 后端沿用现有豆包视觉接入点（`doubao-seed-1-6-vision`，文本 DSL，`<point>x y</point>`），后端 / 接入点可配置。

### 7.4 执行回放（V1 不做，后续路线已定）

V1 先跑通，不做执行回放；后续再加强。路线决策：

- **不采用** web 标准版（`ai_action_merge`）的"**先规划后执行 + 二层坐标缓存**"方案——它把任务先拆成固定步骤再按缓存坐标重放，**灵活性太差**，UI 一变就脆。
- **后续要做，参考 ai-phone 的 V2（状态路标对齐）**：保存成功 Run 的动作轨迹，回放时把"当前页面状态"和"首跑每个动作后的状态路标"对齐；对不齐就等待 / 局部修复 / 判失败，而不是盲目重放旧坐标。这与 AI Web 的截图循环范式天然契合。
- **架构预留（V1 就做）**：执行记录按 §9 完整落库每步的 `action / thought / 坐标 / 前后截图 / token`，形成可回放的证据链；将来直接在历史成功 Run 上做 V2 回放，无需回头改内核。
- **回放身份（待定，见 §15）**：AI Web 没有用例/步骤概念，回放需另定"同一个任务"的认定键（候选：投递的 `caseId` / `runContent` 哈希 / 显式模板 ID）。

## 8. 并发、调度与多 Pod（K8S）

### 8.1 并发 = 可热调的浮动值 N

- N 表示"当前允许多少个浏览器同时接任务"，运行时可调（设 1 / 2 / 3…），存 `t_aiweb_config`，无需重启。
- 调度器从队列按 FIFO 领取 `queued` 任务，拿到并发槽就交给一个 worker；worker：起 `browser_context` → 跑内核 → 出报告 → 发 Webhook → 释放槽。
- 槽满 → 任务在队列等，**谁先释放谁先上**（即"都忙就等，谁先出来跑谁"）。
- 单个 headless Chromium 约 0.3~0.7G 内存，N 按单 Pod 资源设定。

### 8.2 多 Pod 能力：Web 比 ai-phone 容易得多

ai-phone 不做多 Pod，是因为**真机物理绑定**（手机插在某台宿主机/Agent 上），多 Pod 要处理设备归属与机器隔离，非常麻烦。

**Web 没有这个问题**——没有物理设备，浏览器是每个 Pod 进程内本地起的、彼此天然隔离。多 Pod 只需满足两点，且都很便宜：

1. **队列原子领取**：用 PostgreSQL `SELECT ... FOR UPDATE SKIP LOCKED` 领任务，多个 Pod 抢同一队列不会重复领取。
2. **共享存储**：报告 / 素材放共享存储（对象存储 / NFS），任意 Pod 产出的报告都能被前端 / Webhook 访问。

> 你们 K8S Pod 是 1 核 1G，单 Pod 跑 1 个浏览器都偏紧，所以**横向扩 Pod 才是 Web 端获得并发的自然方式**：总并发 = 各 Pod 的 N 之和。

### 8.3 V1 落地建议

- **代码从第一天就按多 Pod 安全写**：队列用 `FOR UPDATE SKIP LOCKED` 领取；报告/素材走可切换的存储抽象（本地卷 ↔ 对象存储）。
- **部署可先单 Pod 起步**（建议 2 核 2G、N=2~3 验证主链路），需要并发时直接加 Pod，无需改架构。
- 进程/Pod 重启恢复：领取任务带心跳/超时，超时未完成的 `running` 任务由其它 Pod 重新领取（V1 简单策略：启动时回收本 Pod 残留）。

## 9. 数据模型（独立 PG 库，`t_aiweb_*`）

| 表 | 含义 | 关键字段 |
|---|---|---|
| `t_aiweb_submission` | 提交批次 | id, name, callback_url, retry_max, state, created_at |
| `t_aiweb_item` | 批次内执行单元（= 一条自然语言任务） | id, submission_id, case_id, case_name, run_content, assets(json), state, attempts, report_url |
| `t_aiweb_run` | 单次实际执行记录 | id, item_id, state, steps, token_usage(json), elapsed_ms, started_at, finished_at, fail_reason, claimed_by, heartbeat_at |
| `t_aiweb_run_step` | VLM 每步（动作、Thought、截图、token、耗时） | id, run_id, step_no, action, thought, screenshot_url, token_usage(json), elapsed_ms |
| `t_aiweb_asset` | 素材文件（执行中可上传的文件） | id, name, path/url, size, mime, created_at |
| `t_aiweb_config` | 平台配置（并发数 N、缓存阈值等，可热调） | key, value |

> 没有用例库 / 项目 / 步骤模板表——AI Web 只有"提交 → 执行 → 记录"+ 素材。
> `t_aiweb_run` 带 `claimed_by` / `heartbeat_at` 是为 §8 的多 Pod 队列领取与超时回收预留。

## 10. 报告方案（我的建议：自绘 HTML 为主，Trace 为辅）

**推荐：自包含 HTML 作为主报告 + 可选附 Playwright Trace 作为深度排障件。**

理由：

| 维度 | 自绘 HTML | Playwright Trace |
|---|---|---|
| 展示 VLM 专属内容（Thought / Action / 决策 / token） | ✅ 完全可控 | ❌ 只有浏览器动作，没有 VLM 思考 |
| 通用可看性 | ✅ 一个 URL 直接打开，适合 Webhook reportUrl + 维护台 | ⚠️ 是 .zip，需 Trace Viewer / 在线查看器 |
| 受众 | 业务 / 运维都能看 | 偏开发深度排障 |
| 成本 | 需自绘（但内核已采集每步截图+Thought+Action，套模板即可） | 近乎白送 |

结论：**主报告自绘**（每步：决策截图 / Thought / Action / 执行后截图 / token / 最终结论），Webhook 回传的就是它的 URL；**同时把 Playwright Trace 作为可选附件**存下来，需要深挖时下载用 Trace Viewer 看。两者兼得。

## 11. 素材文件管理（你的"上传地址 + 执行时自动抓取上传"设想）

- 平台提供**素材库**：一个上传入口 + 文件列表，支持**增删改查**（`t_aiweb_asset` + 共享存储）。
- 投递 item 时可在 `assets` 里声明本次会用到的素材（按名引用素材库里的文件）。
- 执行中当自然语言要求"上传某文件"时，内核 `upload_file` 动作从该 item 的素材清单/素材库按名取文件，调用 Playwright `set_input_files` / `expect_file_chooser` 完成上传，VLM 无需关心系统文件框。
- 即：**先在素材库配置文件，执行时服务自动按引用抓取并上传**，成为内核原生能力。

## 13. V1 范围与不做项

**V1 做：**

- 自然语言投递 → 队列 → N 并发执行 → 自包含报告 → Webhook 回调
- Playwright 内核 + 内置动作集（含 `upload_file`）
- **主动式 LLM 上下文缓存（Responses API + 显式缓存 + 分段）**：移植并启用
- 素材库（增删改查）
- 队列按多 Pod 安全方式实现（`FOR UPDATE SKIP LOCKED` + 共享存储）
- 前端维护台：队列总览 / 执行记录 / 报告查看 / 素材管理 / 并发等配置
- 对外 API + 基础 CLI

**V1 不做（后续迭代，架构预留）：**

- 辅助系统高级能力（审判系统 / 最终断言模型 / 模型介入卡死）——V1 仅保留朴素本地卡死检测 + finished/assert_fail。
- **执行回放缓存**——区别于已启用的 LLM 上下文缓存；V1 不做，后续按 §7.4 走 ai-phone V2 路线（不抄 web 标准版的先规划后执行）。执行记录已预留每步证据，便于将来直接回放。
- VNC / 实时镜像 / 实时进度。

## 14. 未来扩展：AI API（自然语言调接口）

- AI API 与 AI Web **外壳同构**（自然语言投递 → 队列 → 并发执行 → 报告 → Webhook → CLI），但**执行内核完全不同**（一个驱动浏览器，一个发 HTTP 请求 / 串接口 / 断言响应）。
- 策略：V1 **不与 AI Web 物理合并**，但把 AI Web 的"投递 / 队列 / 报告 / 回调 / CLI / 存储"骨架做成**可复用范式**；做 AI API 时复用骨架、只换执行内核（未来可做 monorepo 共享 core + 双执行器）。
- 具体放不放一块，等 AI Web 骨架落地后再定。

## 15. 决策记录与待确认

**已确认：**

| 项 | 结论 |
|---|---|
| 项目形态 | 全新独立项目，前后端分离，单机可部署 |
| 个人 / 公司 | 个人开源版先行，公司版后续迁移（仅差部署配置） |
| 执行框架 | Playwright |
| 数据库 | 复用现有本地 PostgreSQL，新建独立库 `aiweb` |
| 内核 | 以 `ai_vlm_web_segment.py` 为蓝本，保留主动式缓存并预留扩展 |
| 执行回放 | V1 不做；后续走 ai-phone V2（状态路标对齐），不抄 web 标准版先规划后执行 |
| 并发 N | 浮动值，运行时可调（1/2/3…） |
| 多 Pod | 代码按多 Pod 安全写（SKIP LOCKED + 共享存储），部署可先单 Pod |
| 素材 | 素材库（上传 + 增删改查），执行时按引用自动抓取上传 |

3. **共享存储介质**：V1 先本地卷即可，还是直接上对象存储 / NFS（影响多 Pod 时间点）。
