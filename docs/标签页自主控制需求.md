# 浏览器环境感知与自主操作需求

## 背景

当前浏览器自动化任务中，用户在页面 A 点击某个按钮后，网站可能在后台新开页面 B，浏览器可见页面仍保持 A。上传任务素材时，浏览器执行 Agent 已持有可用文件，但 VLM 不一定知道有哪些文件可选。

现有实现会尝试在检测到新标签后自动切到最新标签。这会把“发现浏览器状态变化”和“下一步是否进入 B”混在一起：前者是系统应提供的事实，后者应由 VLM 根据任务自主决定。

本需求只定义通用浏览器能力，不引入任何站点或业务规则，也不要求系统理解 A、B 的业务含义或父子关系。

## 本次目标

让 VLM 获得标签页和任务素材的完整、可操作环境信息，并自行决定留在当前页还是切换标签、何时选择哪个文件上传。

## 功能需求

### 1. 标签状态感知

每次浏览器动作完成后，系统必须感知并维护：

- 当前活动标签；
- 所有仍打开的标签；
- 本次动作中新打开或关闭的标签；
- 每个标签稳定的 `tab_id`，以及用于识别的标题、URL 等基础信息。

当 A 点击后在后台打开 B 时，系统应把事实交给 VLM，而不改变当前页：

```text
当前标签：A
当前画面：A
标签列表：A、B
本轮新增标签：B
```

即使任务文本没有说明“该按钮会打开新标签”，VLM 也必须能知道 B 已经存在。

### 2. VLM 自主决定切换

系统不得因为发现新标签而自动切换到 B。

VLM 根据任务决定下列任一行为：

- 保持在 A，继续完成 A 上的操作；
- 立即切换到 B；
- 在 A 完成后再切换到 B；
- 在多个标签之间往返切换。

系统不根据业务语义决定何时切换，也不默认替 VLM 切回 A。

### 3. 标签操作协议

本次继续复用现有平台软协议，不把迁为标准 Function Tool 作为前置条件。VLM 在获得标签状态后，可通过平台动作自主切换或关闭标签，例如：

```text
PLATFORM_ACTION: switch_tab(tab_id='tab_b')
PLATFORM_ACTION: close_tab(tab_id='tab_b')
```

标签状态由系统自动附带给 VLM，因此不要求 VLM 先调用 `list_tabs()` 才能知道有哪些标签。

其中 `switch_tab` 必须使用稳定的 `tab_id`，不以会变化且没有业务语义的数组序号作为唯一定位方式。

### 4. 结构化执行结果

每个标签工具调用必须返回结构化结果：

- 是否成功；
- 当前标签；
- 最新标签列表或必要的标签变化；
- 切换成功后的当前页截图。

标签不存在、已关闭或不可切换时，必须返回明确错误和最新标签状态，不能静默失败。

### 5. 任务素材感知

每个任务开始时，系统必须向 VLM 提供本任务可用素材清单，至少包括：

- 文件名（作为当前上传动作的选择依据）；
- MIME 类型和大小等必要的识别信息。

素材清单是环境事实，不依赖任务文本是否恰好写出了文件名。VLM 根据当前页面、任务目标和素材清单决定是否上传、何时上传、上传哪一个文件。

当前素材的取得链路应继续保持：服务端生成素材下载地址，浏览器执行 Agent 下载到本机临时目录并维护素材标识到本地路径的映射。VLM 不下载文件，也不接触本地路径。

### 6. 自主上传

上传动作继续复用现有软协议。VLM 在获得素材清单后，按素材名称发出明确的上传动作，例如：

```text
PLATFORM_ACTION: upload_file(name='xxx.pdf')
```

运行时负责将素材名称解析为浏览器执行 Agent 已准备好的本地文件，再完成页面上传。

上传结果必须向 VLM 返回：

- 使用了哪个素材；
- 是否成功；
- 失败原因（素材不存在、当前页面没有可用上传控件等）；
- 上传后的页面截图和必要页面状态。

## 统一原则

标签和素材遵循同一原则：系统提供事实和操作能力，VLM 自主决定业务动作。

| 环境事实 | VLM 的自主动作 |
| --- | --- |
| 新增了 B，当前仍在 A | 保持 A 或切换到 B |
| 当前有素材 X、Y | 不上传、上传 X、或上传 Y |

系统不根据业务语义自动切换标签，也不根据业务语义自动挑选文件。

## 目标交互时间线

```text
VLM 看到 A
→ VLM 点击 A 上的按钮
→ 浏览器后台打开 B，当前可见页仍是 A
→ 系统反馈：当前 A；新增 B；标签为 A、B
→ VLM 根据任务选择继续 A 或调用 switch_tab(B)
→ 若切换：系统返回当前 B 和 B 的截图
```

## 非目标

- 不为特定业务定义“什么时候必须去 B”或“什么时候必须留在 A”；
- 不要求系统推断 A 与 B 的业务父子关系；
- 不要求本次先改造所有既有浏览器动作。

`open_url`、`new_tab` 等现有软协议动作本次不要求改为标准 Function Tool；后续如有统一 Tool 调用改造，再另行评估，不影响本需求中标签感知、素材感知和自主操作的最小闭环。

## 可靠性与可观测性要求

- 新标签发现不能仅依赖一次短时间轮询；应持续追踪浏览器标签创建和关闭事件；
- 每一步运行记录应保存标签变化、当前 `tab_id` 和工具执行结果；
- 每一步上传记录应保存素材标识、执行结果和失败原因；
- VLM 看到的截图、系统声明的当前标签、工具返回的当前标签必须一致；
- 标签状态变化发现失败、切换失败、目标已关闭、素材不存在和上传控件不存在必须在执行结果中显式暴露。

## 验收标准

1. A 点击后后台打开 B，下一轮 VLM 能得知 B 已新增，同时截图仍为 A；
2. VLM 未调用 `switch_tab(B)` 时，后续操作仍执行在 A；
3. VLM 调用 `switch_tab(B)` 后，后续截图和操作页均为 B；
4. VLM 可从 B 切回 A，且不依赖猜测数组索引；
5. B 已关闭或 `tab_id` 无效时，VLM 能收到失败原因和最新标签列表；
6. 运行记录可追踪 A 点击、新增 B、VLM 是否切换、实际当前页分别发生在何时。
7. 任务文本未写文件名时，VLM 仍能获取可用素材清单并选择正确素材；
8. 上传失败时，VLM 能收到具体原因和上传后的最新页面状态，而不是静默失败。
