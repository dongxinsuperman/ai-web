# AI Web 站点映射与免登

> 解决两类问题：① 用户只说"登录课程系统"却没给网址 → 关键字映射到 URL；
> ② 目标站点需要登录 → 预配置免登信息，执行时自动注入、跳过登录。
> 后台可配（存数据库），执行时自动携带，best-effort（给得了能用就用，给不了就正常登/登不上，属调用方输入问题）。

实现：`models/site.py` · `sites.py` · `api/sites.py` · `kernel/browser.py`/`runner.py`/`prompt.py` · `scheduler/worker.py` · 前端 `pages/Sites.vue`。

## 1. 站点条目（`t_aiweb_site`）

| 字段 | 说明 |
|---|---|
| `name` | 展示名 |
| `keywords` | 关键字，分隔符（`/ ， 、 空格 ;` 等）分割；任一命中 runContent 即视为涉及该站点 |
| `url` | 站点地址 |
| `auth_type` | `none` / `storage_state`(静态快照) / `cookies`(静态) / `login_api`(动态现取，不过期) |
| `auth_payload` | 按类型不同的免登数据（JSON）。内网部署、AI Web 无权限，按确认**不加密** |
| `enabled` | 是否启用 |

维护台「站点 / 免登」Tab 增删改查。

## 2. 执行时如何注入（自动）

worker 在建浏览器前，用 runContent 解析命中站点，然后：

1. **网址簿 → 注入 prompt**：把命中站点的 `关键字 → URL` 拼成"网址簿"，加到 system prompt。任务没给明确网址时，模型按关键字 `open_url` 打开正确地址。
2. **登录态 → 注入浏览器**：
   - `storage_state`：合并命中站点的 storageState（cookies + origins），在**建 context 时**带入 → 打开即已登录。
   - `cookies`：`context.add_cookies(...)` 注入。
   - `none`：不注入，正常去登（账号密码写在 runContent 里由模型登）。

> best-effort：注入失败 / 配置不可用不阻断，任务照常执行（能登就登、登不上就失败）。

## 3. 免登方式

### A. storageState 登录态快照（推荐）

"登录态"= 浏览器里的 cookies + localStorage。登录一次后导出成 JSON，复用即免登。

**获取方式（维护台一键录制）**：
1. 配置站点时点【录制登录态】→ `POST /api/sites/record/start {url}` → 后端弹出**有头浏览器**并打开该网址，返回 `recordToken`。
2. 用户在弹出的浏览器中**手动登录**（账号密码 / 验证码随意）。
3. 回维护台点【保存登录态】→ `POST /api/sites/record/save {recordToken}` → 后端 `context.storage_state()` 导出 JSON 返回并填入字段；保存站点即入库。

> 录制需有显示器环境（本地）。无显示器的 Pod 不能录制，可改为粘贴现成 storageState JSON。

### B. Cookie 注入

`auth_payload = {"cookies": [{"name":"token","value":"...","domain":".x.com","path":"/"}]}`，执行时 `add_cookies`。

### C. 登录接口 login_api（动态现取，推荐 · 不过期）

**为什么需要它**：A/B 是静态快照，token 有有效期，时间一长会失效。login_api **每次执行临时调接口拿新 token 再注入**，永远新鲜、不过期（对标标准平台 `shadow_test_login`，但做成通用配置，不绑定具体系统）。

> **配方是通用的，不绑定任何系统**：绝大多数站点只需 `login_url + payload + token_path + inject`（单步登录）；`me`（二次换值）仅个别系统（如 shadow）需要，可选。shadow 只是其中一个特例。

**精确配方（auth_payload.recipe）**：
```json
{
  "login_url": "登录/取token接口", "method": "POST", "headers": {}, "payload": {测试账号},
  "token_path": "shadowToken",            // 响应JSON里token的点路径
  "token_prefix": "",                      // 可选，如 "Bearer "
  "me": {"url":"可选二次换值接口","header_name":"Authorization","resp_header":"ShadowAuthorization"},
  "inject": {
    "cookies": [{"name":"onionsToken","domain":"可选"}],   // value 自动填 token
    "local_storage": [{"key":"ShadowAuthorization"}]        // value 自动填 token
  }
}
```
执行时：调 login_url → 取 token →（可选 me 二次换值）→ 按 inject 写 cookie/localStorage（合进 context 的 storage_state）→ 导航。每次新鲜，best-effort（失败不阻断）。

**精确 + 智能解析两种录入**：
- **精确**：直接填上面的 recipe JSON。
- **智能解析（agentic 自修正，推荐）**：用大白话描述怎么登录（**不用精准**）→ 点【智能解析】→ `POST /api/sites/compile-auth`：模型出一版配方 → **真跑**（调登录接口）→ 把**登录接口真实响应 + 打开站点后的落点**反馈给模型 → 模型据此**自我修正**（如 token 字段猜错了，看真实响应就改对）→ 再跑 → **直到能登进去**，产出一份**已验证可用**的配方，回填供你微调后保存。
  > 设计取舍：**自修正发生在配置期**（用真实执行反馈迭代，因为没人能把登录描述得 100% 精准）；**执行期只跑这份已验证的精确配方、不再调模型**——确定性、快、可调试。
  > `POST /api/sites/parse-auth` 是早期的"只解析不验证"简易版，已被 compile-auth 取代。

### D. 账号密码

**不在此配置**：直接把账号写进任务自然语言，模型打开登录页输入登录（best-effort）。

## 4. 接口

| 接口 | 说明 |
|---|---|
| `GET/POST /api/sites`、`PUT/DELETE /api/sites/{id}` | 站点 CRUD |
| `POST /api/sites/record/start {url}` | 启动录制（弹有头浏览器），返回 recordToken |
| `POST /api/sites/record/save {recordToken}` | 导出 storageState 并关闭录制浏览器 |
| `POST /api/sites/record/cancel {recordToken}` | 放弃录制 |
| `POST /api/sites/compile-auth {description, url}` | **智能解析（agentic 自修正）**：出配方→真跑→看真实响应/落点→修正→直到登进去，返回已验证配方 |
| `POST /api/sites/verify-auth {recipe, url}` | 验证当前配方：真跑 + 无头打开站点，回报落点 / 是否仍在登录页 |
| `POST /api/sites/parse-auth {description}` | （旧）只解析不验证，已被 compile-auth 取代 |

---

## 5. 安全

- `auth_payload` 含 token / 登录态 / 账号 → 敏感凭据，**加密存储**（团队后端规范强制）、报告/日志**不打印**、建议权限控制。
- 录制会话 10 分钟未保存自动回收。
