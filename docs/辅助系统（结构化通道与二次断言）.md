# AI Web 辅助系统：结构化通道 + 二次断言

> AI Web 的辅助系统是**精简版**：只做「结构化通道判定」+「二次断言」，
> 其余 ai-phone 的辅助能力（中途审判、增强卡死、重试、轨迹缓存）**一律不做**。
> 实现：`kernel/assertion.py` + `kernel/runner.py`。

---

## 1. 设计哲学

主 VLM 负责"看图操作"；辅助系统回答一个问题：**"它真的完成了吗？"**

模型自己吐的 `finished()` **经常高估完成度**（看到大致对的页面就宣布成功）。所以对
**结构化用例**，在 `finished` 后再做一次**独立裁决**：拿「预期结果」+ 当前页面截图复核，
而不是采信模型自述。非结构化用例没有明确预期、断言无意义，直接采纳 `finished`。

---

## 2. 范围

| 能力 | 是否做 | 说明 |
|---|---|---|
| 结构化通道判定 | ✅ | 决定是否启用二次断言 |
| 二次断言（最终裁决） | ✅ | 仅结构化通道触发 |
| 页面稳定检测 | ✅（已在内核） | readyState + DOM 静默 |
| 朴素本地卡死检测 | ✅（已在内核） | 连点/连滚注入提示 |
| 中途审判 / 增强卡死 / 重试 / 轨迹缓存 | ❌ 不做 | 保持精简 |

---

## 3. 结构化通道判定

**规则（确定性、零模型调用）**：runContent **命中 ≥ 2 个四级标签 且 必须含「预期结果」** → 结构化通道；否则自由对话通道。

- 四级标签：`测试标题 / 测试用例 / 前置条件 / 操作步骤 / 测试步骤 / 预期结果`
- 命中后按段头标签切分，取「预期结果」段文本作为断言依据。
- 阈值 `_MIN_LABEL_HITS = 2` 为内核常量（与缓存同理，不进配置）。

代码：`assertion.detect_structured(run_content) -> (structured, hits, expected)`。

---

## 4. 二次断言（最终裁决）

仅当**结构化**且有「预期结果」时，在模型 `finished()` 后触发：

- **输入（严格：双图 + 全局步骤）**：「预期结果」段 + **完整步骤摘要**（每步 序号/动作/思考，用于判断是否真执行到位）+ **附图1=末动作前帧** + **附图2=最终帧** + 主模型 thought（仅背景，不可直接采信）。首步即 finished 无前帧 → 退化单图。
- **调用**：**独立一次性**模型调用（`store=false`、不串 `previous_response_id`），与主执行会话隔离，避免"自我背书"。
- **输出**：强制**严格 JSON** `{"passed": true|false, "reason": "..."}`。
- **判定原则（严格但不吹毛求疵；逐条核对，全部成立才通过）**：
  ① 语义等价：措辞/同控件不同表达/不同载体不挑刺；数值「约/前N」可±1（只判是否达成，不纠文字）；
  ② **关键事实需硬证据**：数值/数量/选中态/开关态/页面名/弹窗态/控件存在性必须在截图有明确证据，**找不到明确证据即不成立**（不给"拿不准"放行）；
  ③ **否定型预期**（不存在/没有/无/未显示）：必须有"已充分查看"证据，无法确认 → 不成立；
  ④ **伪成功（双图）**：自述明显变化但前后两图几乎相同 → 不成立；
  ⑤ **结合步骤摘要**：摘要显示根本没执行到验证所需操作（如要查某菜单却从未点开）→ 证据不足、不成立；
  ⑥ 证据不足 / 被遮挡 / 页面不对 / 与摘要矛盾 → false。
- **处置**：
  - `pass` → 用例 **success**
  - `fail` → 用例 **failed**，`status_reason = assertion_failed`，`fail_reason = 二次断言不通过：<reason>`；**不重试**
  - `skip`（调用/解析/超时异常）→ **回退采纳**主模型 `finished` 结果（不阻塞收尾）

代码：`assertion.run_final_assertion(vlm, expected, thought, final_b64) -> (verdict, reason)`。

## 5. 日志与报告体现

每个 run 在执行记录 / 报告时间线里会多出可见步骤：

- `step 0 [通道判定]`：结构化（命中哪些标签）/ 自由对话；是否将启用二次断言。
- 结构化时，`finished` 后追加 `[二次断言] PASS/FAIL/SKIP：<理由>`（附判定时的截图）。

实测示例：

```
step 0 [通道判定] 结构化通道（命中标签：测试标题, 前置条件, 操作步骤, 预期结果）→ finished 时启用二次断言
step 1 [open_url] ...
step 2 [finished] ...
step 3 [二次断言] PASS：页面标题为"Example Domain"，包含"Example"，符合预期。
```

## 6. 与 ai-phone 的差异 / 改进（重点）

我们要这些能力，但**没有照搬 ai-phone 的实现**，对其设计上的冗余/缺陷做了改进：

| 维度 | ai-phone | AI Web（改进） |
|---|---|---|
| 通道判定 | 多信号严格度评分（7 维）+ 模棱两可时调**审判模型**兜底分类（默认评分入口关闭，等于只用标签，却保留了大量复杂度与一次额外模型调用） | **纯标签计数 + 必须含「预期结果」**：确定性、可解释、**零额外模型调用**、无"模棱两可"分支 |
| 断言触发 | 对所有 `finished` 都调一次裁决（"后续才改成只在结构化触发"） | **一开始就 gate**：仅结构化触发，非结构化**零断言开销** |
| 断言输出 | `PASS / FAIL / SKIP` 文本 | **严格 JSON `{passed, reason}`**，解析更稳、reason 可直接进报告 |
| 断言独立性 | 同链路再调 | **独立一次性调用**（`store=false`、不串会话），避免自我背书 |
| 断言证据 | 末步「前帧 + 后帧」双图 + 动作历史 | **双图（末动作前帧 + 末帧）+ 全局步骤摘要** |
| 断言松紧 | 首跑实时断言已弃用（仅缓存回放保留两层判定）；缓存断言要求关键状态有截图证据、证据不足必 FAIL | **严格但不吹毛求疵**：逐条核对预期 + 关键事实需硬证据 + 否定型专项 + 结合步骤摘要 + 证据不足/与摘要矛盾即 FAIL（语义等价不纠文字） |
| 自由对话断言 | 历史有，现已弃用 | **不做**（仅结构化才二次断言） |
| 失败处置 | 进重试/审判等链路 | 直接 `failed(assertion_failed)`，**不重试**（精简） |

> 常量（`_MIN_LABEL_HITS=2`、断言超时 60s）为内核写死，不对外暴露、不进配置。

## 7. 实现位置

| 文件 | 职责 |
|---|---|
| `kernel/assertion.py` | 结构化判定 / 预期结果抽取 / 二次断言裁判（含 prompt 与 JSON 解析） |
| `kernel/vlm_client.py::oneshot` | 独立一次性模型调用（不入主会话缓存） |
| `kernel/runner.py` | run 开始打"通道判定"步骤；`finished` 时 gate 二次断言并据 verdict 决定终态 |
