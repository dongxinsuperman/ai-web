# AI Web 里程碑与实施计划（历史资料）

> ⚠️ 本文是早期 MVP 的估算和计划记录，包含已经完成或已被 Agent 架构替代的内容；不作为当前功能状态或部署依据。

> 配合《方案设计（总体方案）》与《实施蓝图（技术规格）》。本文给出可执行性结论、
> 工程量拆解、里程碑划分与验收标准，便于排期与分阶段交付。

---

## 1. 可执行性结论

**可执行性：高（低风险）。工程量：小到中等。**

核心原因：最难、最不确定的部分（VLM 决策、上下文缓存、自然语言驱动浏览器）在标准版
**已跑通验证**，本项目是"**抄已验证的逻辑 + 换更好的框架重新组装**"，不是攻关未知。

| 模块 | 性质 | 风险 |
|---|---|---|
| VLM 决策循环 / 解析 / 坐标换算 / 卡死检测 / prompt | 已验证，直接抄 | 极低 |
| 主动式上下文缓存（Responses API + 显式缓存 + 分段） | 已验证，直接抄 | 极低 |
| 动作集逻辑 | 逻辑已验证，执行层换 Playwright 重写 | 低 |
| aiphone 式外壳（投递/队列/Webhook/报告） | 设计已验证，新栈重写 | 低 |
| FastAPI / PG / 调度 / worker 池 / 前端 | 全新但标准套路 | 低 |

> 没有一处属于"没人做过、不知能不能成"。

## 3. 里程碑划分

### M1 — 后端主链路打通（可投递、可执行、出结果）
**范围**：FastAPI 骨架 + PG 表 + 投递/查询 API + 队列调度（含 SKIP LOCKED）+ Playwright 内核（含内置动作 + 上下文缓存）+ 最简文本结果。
**验收**：
- `POST /api/submissions` 投一条 `runContent` → 后台起本地浏览器执行 → `GET` 能查到 `success/failed`。
- 内核能完成"打开 URL → 操作 → finished/assert_fail"闭环。
- 并发 N 生效（同时跑多条），队列满会等。
- 跑一遍"动作探针测试集"（技术规格 §5.8），用真实豆包输出校准别名表与 parser；
  未知动作不崩溃（记录原始输出 + 注入提示重试）。

### M2 — 报告 + Webhook + 取消
**范围**：自包含 HTML 报告（每步截图/Thought/Action/token）+ `_summary` + Webhook（item/submission terminal）+ 取消接口 + reaper 心跳回收。
**验收**：
- 每条执行产出可离线打开的报告 URL。
- 带 `callbackUrl` 投递能收到两类终态回调。
- 运行中可取消；Pod 异常时任务能被回收重排。

### M3 — 素材管理 + 前端维护台 + CLI
**范围**：素材库（上传/增删改查 + upload_file 落地）+ 前端 5 个页面 + CLI（submit/get/cancel/open-report）。
**验收**：
- 上传素材后，`runContent` 中"上传文件"能自动抓取素材完成上传。
- 维护台可看队列、执行记录、报告、改并发数、管素材。
- CLI 可完成投递到看报告全流程。

### M4 — 打包 / 部署 / 压测 / 多 Pod 验证
**范围**：Dockerfile（playwright/python 基础镜像）+ K8S 部署 + 并发与内存压测 + 共享存储切换（多 Pod 时）。
**验收**：
- 单 Pod 跑通；按需扩 Pod，多实例从同队列领取不重复、报告互通。
- 1c1G/2c2G 下确定安全并发 N。

> 优先级建议：**M1 → M2 → M3 → M4**。M1+M2 即"后端可用"，M3 补齐人用界面与素材，M4 上生产。

## 4. 关键风险与对策（少数几点，均有兜底）

| 风险 | 说明 | 对策 |
|---|---|---|
| Playwright 坐标映射 | 0-1000 归一化 ↔ viewport / device_scale_factor | 固定 viewport，截图与坐标同尺寸；半天校准 |
| 1c1G Pod 承载 | 单 Pod 跑 headless Chromium 偏紧 | 实测定 N，靠多 Pod 横向扩（架构已支持） |
| headless 渲染差异 / 反爬 | 个别站点对 headless 敏感 | 退回 headful + Xvfb |
| 长任务截图体积 | 步数多导致报告大 | 截图压缩 / 采样 |
| VLM 接入点限流 | 并发高时 429 | 重试退避 + 并发上限约束 |

## 5. 当前待确认（不阻塞 M1 启动）

1. 单 Pod 并发 N 默认值（建议 2–3）+ 首发单 Pod 还是直接多 Pod。
2. API 鉴权：匿名（内网隔离）还是带 token。
3. 共享存储：V1 先本地卷，还是直接对象存储 / NFS。
4. VLM 接入点 / key：个人开源版脱敏占位，公司版用现有接入点。
5. 素材重名策略：覆盖 or 拒绝。

> 以上 1–5 取默认值即可启动 M1，后续随时可调。
